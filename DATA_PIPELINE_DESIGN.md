## 📊 一、消息队列技术选型 (MQ Tech Selection)

### 1.1 候选方案对比

| 维度 | **Redis (Streams/PubSub)** | **RabbitMQ** | **Kafka** |
|------|----------------------------|--------------|-----------|
| **U盘便携性** | ⭐⭐⭐⭐⭐ (嵌入) | ⭐⭐ (需独立服务) | ⭐ (重量级) |
| **部署复杂度** | ⭐⭐⭐⭐⭐ (极低) | ⭐⭐⭐ (中等) | ⭐ (高) |
| **持久化能力** | ⭐⭐⭐ (有限) | ⭐⭐⭐⭐ (强) | ⭐⭐⭐⭐⭐ (极强) |
| **吞吐量** | ⭐⭐⭐ (中等) | ⭐⭐⭐⭐ (高) | ⭐⭐⭐⭐⭐ (极高) |
| **延迟** | ⭐⭐⭐⭐⭐ (极低) | ⭐⭐⭐ (低) | ⭐⭐ (中) |
| **功能丰富度** | ⭐⭐⭐ (基础) | ⭐⭐⭐⭐⭐ (丰富) | ⭐⭐⭐⭐ (流处理) |
| **AI 场景适配** | ⭐⭐⭐⭐ (缓存+MQ两用) | ⭐⭐⭐ (通用) | ⭐⭐ (日志/大数据) |
| **学习曲线** | ⭐⭐⭐⭐⭐ (极简单) | ⭐⭐⭐ (中等) | ⭐ (陡峭) |

---

### 1.2 最终推荐方案：**Redis 优先 + RabbitMQ 备选**

**推荐组合：**
| 场景 | 推荐技术 | 原因 |
|------|----------|------|
| **默认部署** | **Redis (Streams)** | ✅ 完美U盘便携，无需独立服务<br>✅ 兼具缓存+消息队列功能<br>✅ 部署复杂度最低 |
| **生产/高并发** | **RabbitMQ** | ✅ 企业级可靠性<br>✅ 丰富的路由功能<br>✅ 持久化保证 |
| **大数据/日志** | **Kafka** | ✅ 海量吞吐<br>✅ 流处理生态 |

---

### 1.3 Redis Streams 核心优势 (针对我们的场景)

1. **单一服务双重用途** - 既用作缓存，又用作消息队列
2. **完全嵌入式** - 可以与应用在同一进程，或轻量独立服务
3. **U盘友好** - 数据文件可随U盘移动
4. **消费组** - 支持可靠消费、ACK 机制
5. **持久化** - RDB/AOF 支持

### 3.2 核心数据模型设计 (Schema Definition)

#### 3.2.1 用户与会话 (User & Session)
```sql
-- 用户表
CREATE TABLE users (
    id TEXT PRIMARY KEY,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    metadata JSON
);

-- 会话表
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,
    user_id TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_active TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    state JSON,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

#### 3.2.2 事务与决策 (Transaction & Decision)
```sql
-- 决策记录表
CREATE TABLE decisions (
    id TEXT PRIMARY KEY,
    session_id TEXT,
    input TEXT NOT NULL,
    output TEXT,
    confidence REAL,
    mode TEXT,          -- system1 / system2 / hybrid
    metadata JSON,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (session_id) REFERENCES sessions(id)
);

-- 审计日志表
CREATE TABLE audit_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    event_type TEXT NOT NULL,
    entity_type TEXT,
    entity_id TEXT,
    before JSON,
    after JSON,
    metadata JSON,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```
